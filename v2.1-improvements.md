# Evida Prototype — v2.1 Improvements & Fixes

**Date:** 10 Jun 2026
**Status: ✅ All items delivered (10 Jun 2026).** See the v2.1 commit for the implementation. Notes on what shipped, including deviations from the fix sketches below, are at the end of this doc.
**Scope:** Findings from a review of the last two commits —
`eb15419` (v2 UAT feedback implementation) and `8919dfd` (gear-panel demo controls + real state persistence).

The state-persistence model itself is sound: actions write to `EvidaStore` (localStorage), pages read on load, and the gear panel covers unreachable states. The issues below are gaps where a path through the prototype produces broken or unrealistic output, plus cleanup left over from the toolbar removal.

---

## P1 — Bugs a tester will hit

### 1. `?resume=payment` produces a broken confirmation screen
**File:** `screens/booking.html`
**Repro:** Dashboard "Membership pending" banner → *Pay now* → lands on payment (step 6) → pay → confirmation shows **"Invalid Date, null"** for both appointments and "Randox clinic" as location.
**Why:** the resume path calls `goStep(6)` directly without populating `state.clinic / bloodDate / bloodTime / gpDate / gpTime`. Step 7 renders straight from that empty state. The hold timer is also never started, so the retry timer shows a default 15:00.
**Fix:** before `goStep(6)` in the resume branch, run the same auto-select routine the signup flow uses (first clinic + first valid blood slot + first valid GP slot), silently:

```js
if (flow === 'signup' && resume === 'payment') {
  selectClinic(clinics[0].id);
  // pick first non-past blood slot + first GP slot (see issue 3)
  goStep(6);
}
```

### 2. Book-first path loops the user back into booking
**Files:** `screens/login.html`, `screens/booking.html`
**Repro:** `booking.html?entry=bookfirst` → complete booking + payment → confirmation CTA "Create your Evida account →" → `login.html?after=booking` → signup form → "Create account →" → **lands back at booking step 3** and is asked to book again.
**Why:** `handleCreateAccount()` unconditionally redirects to `booking.html?flow=signup`. The `after=booking` context is dropped.
**Fix:** in `handleCreateAccount()`, check whether a booking already exists:

```js
var target = EvidaStore.booking.isConfirmed() ? 'dashboard.html' : 'booking.html?flow=signup';
window.location.href = target;
```

(Reading the store flag is more robust than threading the `after=booking` param into the form, since `handlePayment()` already calls `booking.confirm()` on the book-first path.)

### 3. Booking auto-select picks a date in the past
**File:** `screens/booking.html`
**Repro:** signup flow → step 3 arrives with **Sat 7 Jun** pre-selected — three days ago.
**Why:** the auto-select uses `bloodSlots[0]` blindly; the slot data starts on 2026-06-07 and today is 2026-06-10.
**Fix (minimal):** pick the first slot `>= today`:

```js
var todayISO = new Date().toISOString().slice(0, 10);
var firstBlood = bloodSlots.find(function(s) { return s.date >= todayISO; });
```

**Fix (better, one-time):** generate `bloodSlots` / `gpSlots` dates relative to `Date.now()` instead of hardcoding June 2026 — the hardcoded dates will *all* be in the past within a week, breaking the prototype's realism permanently. This is the more durable fix and worth doing now.

### 4. "Reset to zero" doesn't fully reset
**File:** `screens/store.js` (`clearAll()`)
**Repro:** complete signup → gear panel → *Reset to zero* → login page shows **"Resume your signup? We found a partial account"** banner. Not a fresh user.
**Why:** `clearAll()` clears booking/questionnaire/wearables/consultations/idVerification, but misses `partialAccount`, `accountSaved`, `profileSaved`, the three onboarding keys, notif prefs, notif count, and dark mode. It also hardcodes only 7 of the ~14 legacy `evida:uat:state:<page>` keys.
**Fix:** replace the per-namespace calls with a prefix sweep — simpler and future-proof:

```js
clearAll: function () {
  try {
    var doomed = [];
    for (var i = 0; i < localStorage.length; i++) {
      var k = localStorage.key(i);
      if (k && (k.indexOf('evida:') === 0 || k.indexOf('evida_') === 0)) doomed.push(k);
    }
    doomed.forEach(function (k) { localStorage.removeItem(k); });
  } catch (_) {}
}
```

(`evida_` covers the two legacy underscore keys: `evida_notif_count`, `evida_dark_mode`.)

### 5. "Identity verification pending" banner never goes away
**File:** `screens/dashboard.html`
**Repro:** (a) fresh user, nothing booked → banner already nags about ID verification; (b) upload ID in profile → return to dashboard → checklist item ticks but the **amber banner still shows**.
**Why:** `#idReminder` has no default `display:none` and `initStatusBar()` never touches it. The old UAT states used to toggle it; the rewrite dropped that.
**Fix:** in `initStatusBar()`:

```js
document.getElementById('idReminder').style.display =
  (booked && !idDone && !postConsult) ? '' : 'none';
```

## P2 — Realism / journey gaps

### 6. Post-consult state has no route from dashboard to the results
**File:** `screens/dashboard.html`
**Repro:** gear panel → *Post-consult* → lands on post-consult.html ✓. But navigate to the dashboard: badge says "Your results are ready", yet nothing on the page links to `post-consult.html` — the appointments rows are static and still say "Results pending".
**Why:** the old `post-consult` UAT state injected a "Results ready — view report →" link into the appointment row; the rewrite kept the badge/subtitle logic but dropped the row update.
**Fix:** in the `postConsult` branch of `initStatusBar()`, restore the row update (change the GP consult row's meta to "Results ready — view report →", point its action at `post-consult.html`) and/or make `#sbBadge` clickable in that state.

### 7. Payment randomly fails 20% of the time
**File:** `screens/booking.html` (`handlePayment()` — `Math.random() > 0.2`)
This is good for *testing* the error state but bad for *demoing*: a stakeholder walkthrough fails 1 in 5 runs at the most important moment.
**Fix:** make success the default and the failure deterministic — e.g. selecting the second saved card (or a "Test failure" payment row) triggers the error path. Alternatively add a "Fail next payment" toggle to the gear panel. Keeps the error state testable on demand without ambushing demos.

### 8. GP minimum date is hardcoded, contradicting the stated rule
**File:** `screens/booking.html`
The GP grid disables dates `< '2026-06-12'` with the tooltip "Your results need 4–5 working days" — but the threshold is fixed, not derived. With blood auto-selected on 11 Jun, a GP consult on 12 Jun is allowed, visibly contradicting the tooltip.
**Fix:** compute the GP minimum as `bloodDate + 5 working days` and re-render the GP grid when the blood date changes. (Falls out naturally if slot dates become relative per issue 3.)

### 9. Greeting ignores the name typed at signup
**File:** `screens/dashboard.html`
`#sbGreeting` is hardcoded "Welcome back, James". Since `8919dfd`, `handleCreateAccount()` saves the real name via `account.save()` — one line makes the prototype feel personal when a tester types their own name:

```js
var acct = EvidaStore.account.saved();
if (acct && acct.name) sbGreeting.textContent = 'Welcome back, ' + acct.name.split(' ')[0];
```

### 10. Gear-panel journeys don't seed the account
**File:** `screens/_uat.js`
"Baseline booked" / "Pre-consult complete" / "Post-consult" jump into the app with no account saved — combined with issue 9 this would show a blank greeting, and the login resume banner state is inconsistent. Add `EvidaStore.account.save({ name:'James Chen', email:'james.chen@gmail.com', ... })` to a shared seed step in each journey.

### 11. Checklist click can overwrite the post-consult badge
**File:** `screens/dashboard.html`
`updateChecklist()` always sets `#sbBadge` to "You're X% ready…". In the post-consult state, manually toggling a checklist item resets the badge from "Your results are ready". Minor — guard `updateChecklist()`'s badge write with the `postConsult` flag (or hide the checklist entirely post-consult, which is arguably more realistic anyway).

## Delivery notes (added after implementation)

All 13 items shipped, plus one **P0 found during implementation**:

- **14. (P0) dashboard.html loaded `store.js` *after* its inline script.** Every `EvidaStore` call in `initStatusBar()` threw `ReferenceError` on load, so none of the state-driven dashboard logic from `8919dfd` ever ran (the failures were invisible because neighbouring IIFEs swallowed errors in try/catch). Fixed by moving `store.js` + `_nav-helpers.js` above the inline script. All other pages were verified to load scripts in the correct order.

Deviations from / additions to the sketches above:

- **Issue 3** took the "better" path: `bloodSlots`/`gpSlots` are now generated relative to today (blood = next 5 non-Sunday days; GP = 8 weekdays from blood-min + 5 working days), so the prototype never goes stale.
- **Issue 8**: `gpMinDate()` derives from the selected blood date; changing the blood date re-renders the GP grid and invalidates a now-too-early GP selection. The step-4 info block dates are dynamic too.
- **Issue 7** became a gear-panel item: **"Fail next payment"** — a one-shot toggle (`evida:demo:fail-payment`) consumed by `handlePayment()`. Default payments always succeed.
- **Booking details are now persisted**: `booking.confirm({clinic, bloodDate, bloodTime, gpDate, gpTime})` stores JSON (legacy `'true'` still reads as confirmed) and `booking.details()` reads it back. The dashboard appointment rows render from it; a fresh user sees a "No appointments yet" empty state instead of hardcoded June rows.
- **Issue 11**: post-consult, the checklist card is hidden entirely (rather than guarding the badge — though the guard was added too) and the appointments section retitles to "Recent appointments".
- Extra copy/state fixes made along the way: review-cart slot rows (`cartBloodSlot`/`cartGpSlot`) now show the real selection; the confirmation fasting reminder uses the real blood-test time; the stale "July availability: Tuesdays and Fridays only" line was replaced with evergreen copy; the signup screen shows booking-aware copy when arriving via `?after=booking`; `handleCreateAccount()` clears the partial account once the real one is saved; post-consult unlocks the "Blood tests" and "Health score" stat cards.
- Verified with `node --check` over every inline script + runtime smoke tests of `store.js` (legacy confirm, details round-trip, `clearAll` sweep) and the slot-generation logic.

## Verification after fixes

1. **Resume payment:** dashboard `?pending=payment` → Pay now → step 6 → pay → confirmation shows real clinic, dates, times.
2. **Book-first:** `booking.html?entry=bookfirst` → book + pay → create account → lands on **dashboard** at 33% (not booking).
3. **Reset:** signup → reset via gear → login shows **no** resume banner; dashboard (if visited) is fully fresh.
4. **ID banner:** fresh user → no amber ID banner; book → banner appears; verify ID → banner gone.
5. **Dates:** step 3 auto-selects a date ≥ today; GP grid only offers dates ≥ blood date + 5 working days.
6. **Post-consult:** gear → Post-consult → dashboard → results reachable via appointment row link.
7. **Payment failure:** triggerable on demand; default path never fails.
